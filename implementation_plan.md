# 🧹 깃허브 포트폴리오 클린업 및 폴더 구조 개편 계획

깃허브를 완벽하고 깔끔한 포트폴리오용으로 만들기 위해, 논문이나 대본 등 지저분한 파일은 로컬(본인 PC)에만 남기고 깃허브 기록에서는 완전히 삭제(세탁)하는 계획입니다.

## 1. 📂 폴더 구조 개편 계획 (한글/영문 분리)

공식 문서들이 섞이지 않도록 `ko`와 `en` 폴더로 분리하여 매우 전문적인 오픈소스 구조를 만듭니다.

```text
📦 LLM_Gateway_Project
 ┣ 📜 README.md (한국어 대문 - 필수)
 ┣ 📜 README_EN.md (영문 대문 - 최상단 배치 권장)
 ┣ 📂 docs/
 ┃ ┣ 📂 ko/ (한국어 문서 전용)
 ┃ ┃ ┣ 📜 ARCHITECTURE.md
 ┃ ┃ ┣ 📜 API_REFERENCE.md
 ┃ ┃ ┗ ... (총 4개)
 ┃ ┣ 📂 en/ (영문 문서 전용)
 ┃ ┃ ┣ 📜 ARCHITECTURE_EN.md
 ┃ ┃ ┣ 📜 API_REFERENCE_EN.md
 ┃ ┃ ┗ ... (총 4개)
 ┃ ┗ 📂 captures/ (공식 문서에 쓰이는 이미지들 - 유지 필수)
 ┣ 📂 app/ , 📂 frontend/ (소스 코드 - 유지)
 ┣ 📂 tests/ (테스트 코드 - 유지 강력 추천 ⭐)
 ┃
 ┗ 📂 local_materials/ (새로 만들 폴더 ⚠️)
   ┣ 📜 최종본.pdf, 논문.md, 발표대본.md 등등
   ┗ 👉 이 폴더는 본인 PC에만 남고 깃허브에는 절대 올라가지 않도록 처리됩니다.
```

## 2. 💡 테스트 스크립트(`tests/`)에 대한 의견

> [!TIP]
> **결론부터 말씀드리면, `tests/` 폴더는 무조건 깃허브에 올리셔야 합니다!**

포트폴리오 심사관이나 면접관이 코드를 볼 때, **"이 개발자가 스스로 짠 코드의 안정성을 검증할 줄 아는가?"**를 매우 중요하게 봅니다. 테스트 코드가 존재한다는 것 자체가 "TDD(테스트 주도 개발)에 대한 이해도가 있고, 신뢰성 있는 보안 코드를 짤 줄 안다"는 강력한 무기가 됩니다. 따라서 삭제하지 않고 유지하는 것을 강력히 추천합니다.

## 3. 🧼 깃허브 "삭제 기록" 없이 완벽하게 지우는 방법

깃허브에서 단순히 파일을 지우고 Push하면 "A 파일 삭제됨" 이라는 과거 기록(Commit History)이 영원히 남습니다. 이를 흔적조차 남지 않게 깔끔하게 지우기 위해 **히스토리 초기화**를 진행합니다.

1. `local_materials/` 폴더를 만들어 지저분한 자료(논문, 대본 등)를 모두 몰아넣습니다.
2. `.gitignore` 파일에 `local_materials/` 를 추가하여 깃허브가 이 폴더를 투명인간 취급하도록 만듭니다.
3. 프로젝트 내의 숨겨진 `.git` (히스토리 저장소) 폴더를 삭제하여 과거의 모든 기록을 폭파시킵니다.
4. 깨끗하게 정리된 현재 상태를 기준으로 **"Initial Commit"**을 새로 생성하여 깃허브에 **강제 덮어쓰기(Force Push)** 합니다.
5. 결과: 본인 PC에는 논문/대본이 그대로 남아있지만, 깃허브에는 처음부터 완벽하게 정리된 공식 문서와 소스코드만 존재했던 것처럼 보여집니다.
