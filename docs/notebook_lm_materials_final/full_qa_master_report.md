# 🛡️ 최종 Q&A 마스터 보고서 (발표 대비용)

본 문서는 프로젝트 심사 시 교수님 및 외부 심사위원이 던질 수 있는 **모든 난이도의 예상 질문 14가지**를 4개의 카테고리로 집대성한 최종 방어 대본입니다.

## 📌 카테고리 1: 아키텍처 및 시스템 설계

### Q1. 왜 굳이 정규식과 LLM을 하이브리드로 쓰는가? 정규식만 쓰면 안 되는가?
**A.** "정규식은 처리 속도가 1ms 이하로 매우 빠르지만, 주민번호나 사원번호 같은 **'패턴이 있는 정형 데이터'**만 잡을 수 있습니다. 반면, 신소재 배합비나 공정 온도 같은 **'비정형 기밀'**은 패턴이 없기 때문에 문맥을 파악해야만 탐지가 가능합니다. 따라서 빠르고 명확한 것은 정규식(Phase 1)이 직렬로 처리하고, 문맥이 필요한 것은 LLM(Phase 3)이 맡는 역할 분담이 아키텍처적으로 가장 효율적이라고 판단했습니다."

### Q2. 기밀을 다 난수로 마스킹해버리면, 외부 AI가 어떻게 그 질문에 답변을 할 수 있는가?
**A.** "가장 핵심적인 질문입니다. 저희는 외부 AI에게 '가려진 기밀을 풀어라'라고 요구하는 것이 아닙니다. 외부 AI는 치환된 난수 토큰(`__MASK_PER_...`)을 마치 사람 이름 같은 **'고유명사'로 취급**하여 문장 구조(Syntax)를 유지한 채 일반적인 요약이나 번역을 수행합니다. 이후 응답이 돌아오면, 저희 게이트웨이가 Redis DB를 조회하여 원래의 기밀 데이터로 완벽하게 '역치환'해주기 때문에 아무런 모순이 발생하지 않습니다."

### Q3. 시중에도 개인정보를 차단하는 기존 DLP(유출 방지 솔루션)나 방화벽이 많은데, 이 시스템만의 차별성은 무엇인가?
**A.** "기존 네트워크 방화벽(WAF)은 SQL 인젝션은 막아도, 사람의 언어로 교묘하게 작성된 **'LLM 프롬프트 인젝션'**은 구조적으로 탐지할 수 없습니다. 또한 기존 DLP는 정형 데이터만 잡습니다. 저희 게이트웨이는 기존 보안망을 뜯어고치는 것이 아니라, 기존 방화벽 뒤에 추가되어 **'AI 특화 위협'만을 전담하여 2차 방어하는 추가적인 보안 계층(Layer)**이라는 점에서 확실한 차별성과 수요가 있습니다."

## 📌 카테고리 2: 인공지능 및 프롬프트 엔지니어링

### Q4. 수많은 오픈소스 LLM 중 굳이 'Qwen2.5(알리바바)' 모델을 선택한 이유는?
**A.** "시행착오 V3 단계에서 Meta의 Llama-guard3를 테스트해 보았으나, 서구권 중심의 데이터로 학습되어 한국어 우회 공격 탐지율이 현저히 떨어졌습니다. 반면 Qwen2.5는 아시아 언어 추론 능력이 동급 파라미터 대비 최고 수준이며, 특히 저희가 시스템 프롬프트로 요구한 JSON 응답 포맷을 완벽하게 준수하는 **Instruction Following(지시어 수행) 능력이 매우 탁월**하여 최종 엔진으로 채택했습니다."

### Q5. Phase 2(의도 판정)와 Phase 3(기밀 추출)를 하나의 프롬프트로 묶지 않고 굳이 나눈 이유는?
**A.** "초기에는 API 호출 비용(시간)을 아끼기 위해 두 작업을 한 번에 시도했습니다. 하지만 소형 모델(7B)에게 '공격 의도도 판단하고 동시에 기밀도 찾아라'라고 다중 지시를 내렸더니 모델이 헷갈려하는 **'Task Confusion(태스크 혼동)'** 현상이 발생했습니다. 이를 해결하기 위해 역할을 완전히 분리하여 정확도를 높이고, 대신 `asyncio.gather`를 통해 **비동기 병렬로 동시 실행**하여 속도 지연을 상쇄했습니다."

### Q6. 제미나이(외부 AI)가 응답을 생성할 때 난수 토큰을 대소문자로 바꾸거나 기호를 빼먹으면(환각) 역치환이 실패하지 않는가?
**A.** "아주 예리하신 지적입니다. 이를 방어하기 위해 두 가지를 적용했습니다. 첫째, 토큰을 `__MASK_...__` 형태로 만들어 최신 LLM들이 '건드리면 안 되는 코드 블록'으로 인식하게 하는 **프롬프트 앵커링**을 적용했습니다. 둘째, 로컬에서 역치환 시 대소문자를 무시하고 특수기호 유실을 허용하는 **퍼지 매칭(Fuzzy Matching) 정규식**을 사용하여 복원 탄력성을 확보했습니다."

### Q7. 네거티브 리스트(과마스킹 방지)가 프롬프트에 하드코딩되어 있는데, 실운영 시 유연성이 떨어지지 않는가?
**A.** "PoC 단계에서는 안정성을 위해 하드코딩했습니다. 하지만 실제 기업 배포 시에는 이 리스트를 사내 DB(PostgreSQL)에 저장해 둡니다. 직원이 질문을 보낼 때마다 게이트웨이가 **DB에서 최신 네거티브 리스트를 가져와 시스템 프롬프트를 동적으로 조립(Dynamic Prompting)**하므로, 코드 수정 없이 관리자 화면에서 단어만 추가하면 즉각 반영되는 유연한 구조입니다."

## 📌 카테고리 3: 성능 및 벤치마크 지표

### Q8. 종합 방어율이 78.33%라면 결국 위협의 22%는 놓친다는 뜻인데, 시스템으로 쓸 수 있나?
**A.** "미탐의 22%를 분석해 보면 키릴 문자를 섞어 쓰는 등 소형 7B 모델이 문맥을 놓친 경우입니다. 즉, 이는 **엔진 추론 능력의 한계이지 저희가 설계한 하이브리드 아키텍처의 결함이 아닙니다.** 반대로 정밀도가 99.86%라는 사실은 저희의 방어 뼈대(Fail-Closed 정책)가 완벽하게 통제되고 있음을 증명합니다. 인프라를 늘려 70B 대형 모델로 교체하면 방어율은 즉시 90% 이상으로 상승할 것입니다."

### Q9. 한국어 오탐 0건, 정밀도 99.86%는 어떻게 나온 수치인가? LLM이 JSON 출력을 실패해서 에러로 차단된 비율도 섞여 있는 것 아닌가?
**A.** "오류와 실제 차단을 분리하기 위해 **Temperature를 0.0으로 고정하여 결정론적 환경**을 구성했습니다. 1,200건의 로그를 전수 조사한 결과, Qwen 모델의 순수 포맷 붕괴(JSON 오류)로 인한 Fail-Closed 발동률은 1% 미만이었습니다. 따라서 정밀도 99.86%는 착시가 아니라 철저히 통제된 추론의 결과입니다. 실운영 시에는 가용성을 위해 1~2회 내부 재시도(Retry)하는 로직을 적용할 예정입니다."

### Q10. 테스트에 사용한 1,200건의 데이터는 직접 작성한 것인가?
**A.** "객관성을 위해 보안 커뮤니티의 탈옥(Jailbreak) 프롬프트 데이터셋과, 일반적인 업무 상황(번역, 코드 리뷰 등)을 가정한 시나리오들을 조합하여 자체적인 자동화 QA 데이터셋을 구축했습니다. 한국어 600건, 영어 600건으로 균등하게 나누어 모델의 다국어 처리 능력을 교차 검증했습니다."

## 📌 카테고리 4: 실운영 및 엔터프라이즈 환경 (Scale)

### Q11. 단일 PC 환경에서 테스트했는데, 직원 30명이 동시에 질문하면 CPU가 고갈되어 엄청나게 느려지지 않겠는가? (QPS 지적)
**A.** "맞습니다. 현재 벤치마크의 1~3초 지연은 '단일 사용자' 기준이며, 본 프로젝트는 'GPU 없이 구동 가능한가?'를 검증한 PoC입니다. 실제 엔터프라이즈 환경에서 다중 사용자 트래픽을 처리할 때는 **vLLM 모델 서빙 엔진**을 도입하여 다중 요청을 묶어 처리(Continuous Batching)하고, 도커 컨테이너를 쿠버네티스로 **수평 확장(Scale-out)**하도록 아키텍처를 스케일업 할 계획입니다."

### Q12. 32GB RAM 노트북을 전용으로 계속 켜둬야 한다면 어떻게 '도입 비용이 0원'인가?
**A.** "비용 0원이란, 수천만 원에 달하는 고가의 GPU 서버 인프라 투자가 필요 없는 **'초기 진입 장벽'의 부재**를 뜻합니다. 최근 기업 부서에는 고사양 유휴 PC나 저사양 NPU가 탑재된 소형 미니 PC(어플라이언스)가 흔히 존재하므로, 이를 게이트웨이 전용으로 할당하면 하드웨어 추가 지출 없이 즉각적인 전사 AI 보안망을 구축할 수 있다는 경제적 이점을 강조한 것입니다."

### Q13. 시스템 프롬프트도 긴데, 직원이 1만 줄짜리 코드를 넣으면 LLM의 Context Window가 초과되어 시스템이 뻗지 않는가?
**A.** "매우 현실적인 지적입니다. 7B 모델의 한정된 토큰 제한을 우회하기 위해, 데이터가 일정 길이를 초과하면 게이트웨이 단에서 **청킹(Chunking) 로직**이 발동됩니다. 거대한 텍스트를 N개의 덩어리로 잘라서 병렬로 마스킹을 수행한 뒤 다시 순서대로 재조립하여 내보내는 분산 로직이 적용되어 OOM(Out of Memory) 현상을 방지합니다."

### Q14. 결국 장기적으로 70B 모델을 쓰려면 GPU 서버를 사야 하니까 '비용 강점'과 모순되는 것 아닌가?
**A.** "저희 시스템의 타겟 고객은 이원화되어 있습니다. 초기 진입하는 중소규모 팀은 무비용으로 CPU 모델을 쓰고, 70B 모델 고도화가 필요한 금융/대기업 등은 **이미 사내에 자체 Private GPU 팜을 구축해 둔 경우가 대부분**입니다. 따라서 그들이 이미 투자해 둔 기존 하드웨어 인프라 자원 위에 저희 게이트웨이만 가볍게 소프트웨어적으로 스케일업해서 얹을 수 있다는 확장의 유연성입니다."
