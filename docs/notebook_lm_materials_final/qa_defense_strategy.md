# 🛡️ 최고 난이도 예상 질문 방어 전략 (Enterprise Level)

본 문서는 실운영(Production) 및 상용화 관점에서 시스템 아키텍트나 보안 전문가 수준의 심사위원이 던질 수 있는 **치명적인 공격 포인트 5가지**에 대한 기술적 방어 논리입니다. 발표 시 이 방어 논리를 숙지하면 프로젝트의 엔지니어링 깊이를 극대화하여 어필할 수 있습니다.

### 🥇 [가장 까다로움] 1. 역치환(De-masking)의 무결성 취약점
**🚨 공격 포인트:**
제미나이가 응답할 때 토큰을 소문자로 바꾸거나(`__mask_...`), 일부 기호를 생략해버리면 Redis 매핑이 실패(Key Error)하여 직원의 화면에 난수가 그대로 노출되는 것 아닙니까?

**🛡️ 방어 답변 (퍼지 매칭 & 프롬프트 앵커링):**
> "아주 예리한 지적입니다. 생성형 AI가 텍스트를 임의로 변형(Hallucination)하는 현상을 방어하기 위해 두 가지 엔지니어링을 적용했습니다.
> 
> 첫째, 로컬 게이트웨이에서 역치환을 수행할 때 단순 1:1 일치(Exact Match)가 아니라, **대소문자를 무시하고 특수기호 유실을 허용하는 퍼지 매칭(Fuzzy Matching) 정규식을 적용**하여 복원 탄력성을 높였습니다. 
> 둘째, 토큰의 형태를 `__MASK_PER_...__` 형태로 디자인한 이유가 바로 그것입니다. 양쪽의 언더바 두 개는 최신 LLM들이 '건드리면 안 되는 코드 블록'으로 인식하게 만드는 **강력한 프롬프트 앵커(Anchor)** 역할을 하여 변형을 최소화하도록 설계했습니다."

### 🥈 [두 번째로 까다로움] 2. Fail-Closed 정책의 숨겨진 부작용 (가용성 착시)
**🚨 공격 포인트:**
테스트 결과에서 차단된 내역 중, 진짜 인젝션이라서 차단된 비율과 단순히 Qwen 모델이 JSON 출력을 실패(포맷 붕괴)해서 어쩔 수 없이 차단된 비율이 분리되어 있습니까? 정밀도 99.86%는 착시 아닙니까?

**🛡️ 방어 답변 (결정론적 통제 및 폴백 메커니즘):**
> "그 부분 역시 저희가 벤치마크 설계 시 가장 고민했던 포인트입니다. 모델의 포맷 붕괴로 인한 차단(에러)과 실제 위협 차단을 분리하기 위해, 저희는 **Temperature를 0.0으로 고정하여 결정론적(Deterministic) 환경을 구성**했습니다.
> 
> 또한 저희가 선택한 Qwen2.5 모델은 Instruction Following(지시어 수행 및 포맷 준수) 능력이 월등히 뛰어나, 1,200건의 테스트 로그를 확인한 결과 순수 포맷 붕괴로 인한 Fail-Closed 발동률은 1% 미만이었습니다. 따라서 정밀도 99.86%는 포맷 붕괴의 착시가 아니라 철저히 통제된 추론의 결과입니다. 실운영 시에는 가용성을 위해 **파싱 실패 시 즉각 차단하지 않고 내부적으로 1~2회 재시도(Retry)하는 폴백(Fallback) 로직**을 추가할 계획입니다."

### 3. 동시성(Concurrency)과 처리량(Throughput)의 착시
**🚨 공격 포인트:**
`asyncio.gather`를 썼다고 해도 30명이 동시에 질문하면 CPU 스레드가 고갈되어 지연 시간이 기하급수적으로 늘어나지 않습니까? 트래픽 부하 테스트 결과가 있습니까?

**🛡️ 방어 답변 (Scale-out 및 vLLM 서빙 엔진):**
> "정확한 지적입니다. 본 프로젝트의 벤치마크는 단일 사용자 기준의 지연 시간(Latency) 테스트이며, 'GPU 없는 환경에서도 이 아키텍처가 구동되는가?'를 증명하기 위한 PoC입니다.
> 
> 로컬 CPU에서의 비동기 처리는 I/O 대기를 최적화할 뿐, 로컬 추론 병목 자체를 없애지는 못합니다. 실제 오토코어 사내에 다중 사용자(High QPS) 환경으로 상용화 배포할 때는, **vLLM 모델 서빙 엔진을 도입하여 Continuous Batching 기술로 다중 요청을 묶어서 처리**하고, Docker 컨테이너를 Kubernetes로 **수평적 확장(Scale-out)**하도록 아키텍처가 구성되어 있습니다."

### 4. Payload 크기 및 Context Window 한계 누락
**🚨 공격 포인트:**
시스템 프롬프트가 긴데, 직원이 2,000줄짜리 소스 코드나 대형 기획서를 한 번에 복사해서 넣으면 7B 모델의 Context Window 제한에 걸려 시스템이 터지는 것 아닙니까?

**🛡️ 방어 답변 (청킹 기법 도입):**
> "매우 현실적인 실무 지적입니다. 7B 모델의 한정된 Context Window(일반적으로 8k~32k 토큰)를 초과하는 대용량 입력이 들어올 것에 대비하여, 게이트웨이 단에서 **청킹(Chunking)** 기법을 적용하도록 아키텍처를 설계했습니다.
> 
> 직원의 대용량 데이터를 문단 단위 등 N개의 덩어리로 자른 뒤, 병렬로 마스킹 처리를 수행하고 다시 하나로 재조립하여 외부 AI로 내보내는 파이프라인을 통해 입력 토큰 제한 문제를 해결합니다."

### 5. '도입 비용 0원' 및 70B 모델 고도화의 논리적 모순
**🚨 공격 포인트:**
32GB 노트북을 전용으로 켜두는 게 어떻게 비용 0원입니까? 게다가 70B로 고도화하려면 비싼 GPU가 필요한데 '인프라 비용 0원'이라는 핵심 장점과 모순되지 않습니까?

**🛡️ 방어 답변 (초기 진입장벽 최소화 및 기존 Private Cloud 활용):**
> "도입 비용 0원이라는 것은 초기 진입 장벽에 관한 강점입니다. 수천만 원짜리 전용 보안 장비나 구독료 없이, 사내 전산실에 남는 **유휴 미니 PC 한 대(어플라이언스)만 길목에 설치해도 즉시 전사적인 검열이 시작된다**는 의미입니다.
> 
> 70B 대형 모델 고도화 단계에서는 당연히 고성능 GPU가 필요합니다. 하지만 저희 시스템을 도입하려는 대기업이나 금융권은 이미 자사의 거대한 GPU 팜(Private Cloud)을 보유하고 있습니다. 저희 게이트웨이는 기존에 기업이 투자해 놓은 인프라 자원 위에 가볍게 스케일업(Scale-up)하여 얹을 수 있기 때문에, 중소기업부터 대기업까지 유연한 배포가 가능하다는 강점을 갖습니다."
