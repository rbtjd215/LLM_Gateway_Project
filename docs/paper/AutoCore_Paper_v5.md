# 기업 환경을 위한 로컬 LLM 기반 하이브리드 AI 보안 게이트웨이 설계

**A Hybrid AI Security Gateway Based on Local LLM for Enterprise Environments: Architecture Design**

## 요약

생성형 AI의 기업 도입이 가속화됨에 따라, 임직원의 비의도적 기밀 유출과 프롬프트 인젝션이라는 두 가지 보안 위협이 심화되고 있다. 본 연구는 기존 DLP의 문맥 파악 한계와 네트워크 전면 차단의 구조적 역설을 분석하고, 이를 해결하기 위한 로컬 LLM 기반 하이브리드 AI 보안 게이트웨이 아키텍처를 제안한다. 제안 시스템은 패턴 기반 고속 필터, LLM 의도 분류, 생성형 마스킹의 3단계 파이프라인을 통해 기밀 데이터의 외부 노출 위험을 최소화하면서도 AI 활용을 보존하는 구조를 설계하였다.

**Key Words**: AI Security Gateway, Data Loss Prevention(DLP), Prompt Injection Defense, Local LLM, Dynamic Masking

## 1. 서 론

생성형 AI의 업무 활용이 급속히 확산되면서, 기업 보안 환경에 새로운 유형의 위협이 대두되고 있다. Cyberhaven(2024)의 보고에 따르면, 임직원이 AI 도구에 입력한 데이터 중 27.4%가 기밀 정보로 분류되었으며, 이는 전년 대비 156% 증가한 수치이다<sup>[1]</sup>. AI 도구에 입력되는 기업 데이터의 총량 역시 1년간 485% 증가하여, 비의도적 기밀 유출의 규모가 급격히 확대되고 있다. 또한 동 보고서에 따르면 68%의 직원이 개인 계정을 통해 AI 도구를 사용하며 기업 보안 체계를 우회하는 것으로 나타났다<sup>[1]</sup>.

이러한 위협이 현실화된 대표적 사례로 2023년 삼성전자 사건이 있다. 삼성전자 DS(반도체) 부문에서는 생성형 AI 허용 직후 임직원이 ChatGPT에 반도체 설비 소스코드, 수율 분석 코드, 사내 회의록을 입력하는 3건의 기밀 유출 사고가 발생하였다<sup>[2]</sup>. 이에 삼성전자는 AI 서비스 접근을 전면 차단하는 조치를 취하였으나, 이러한 전면 차단 방식은 근본적인 해결책이 되지 못한다. 사내망에서 AI 접속을 막더라도 직원들은 개인 스마트폰이나 모바일 네트워크를 통해 AI를 계속 사용하게 되며, 이 경우 기밀 데이터가 회사의 감시·통제 영역을 완전히 벗어나 유출 경로조차 추적할 수 없는 '섀도우 AI(Shadow AI)' 상태에 빠지게 된다. 즉, 전면 차단은 문제를 해결하는 것이 아니라 **유출 사실 자체를 인지할 수 없게 만들어, 감사 기록(언제, 누가, 무엇을 유출했는지에 대한 로그)조차 남길 수 없는 완전한 통제 불능 상태를 초래하는 역설적 결과**를 낳는다.

한편, OWASP(2025)는 LLM 애플리케이션 보안 위협 Top 10에서 프롬프트 인젝션(Prompt Injection)을 1위, 민감 정보 노출(Sensitive Information Disclosure)을 2위로 지정하였다<sup>[3]</sup>. 악의적 명령어를 자연어 입력에 삽입하여 LLM의 동작을 탈취하는 프롬프트 인젝션은, AI 시스템 고유의 구조적 취약점으로서 기존 보안 체계가 전혀 대응하지 못하는 새로운 공격 벡터이다<sup>[4]</sup>. 이는 기밀 유출과 더불어 AI 시스템 자체의 악용 가능성이 실질적 위협으로 부상하였음을 의미한다.

결국 기업에게 필요한 것은 AI를 전면 차단하는 것이 아니라, **기밀은 보호하되 AI 사용은 안전하게 허용하는 중간 지점의 보안 아키텍처**이다. 일부 대기업은 기밀 유출을 막기 위해 사내망 전용 폐쇄형 LLM을 구축하기도 하나, 업무 고도화를 위해 압도적 성능을 가진 외부 상용 AI(GPT-4 등)에 대한 임직원의 수요는 지속되며, 자체 모델 구축 여력이 없는 중소기업은 외부 AI 의존도가 더욱 높다. 따라서 기업 규모와 자체 모델 보유 여부를 불문하고, 외부 AI 접속 지점을 통제하는 보안 게이트웨이는 필수적이다. 본 연구는 이러한 문제의식에 기반하여, 사내 로컬 환경에서 구동되는 LLM을 활용하여 기밀 데이터를 자동으로 탐지·마스킹하고, AI 응답 시 원본을 복원하는 하이브리드 보안 게이트웨이 아키텍처를 제안한다.

## 2. 본 론

### 2.1 기존 보안 기술의 분석 및 한계

#### 2.1.1 기존 DLP(Data Loss Prevention) 솔루션

DLP(데이터 유출 방지)는 기업 내부의 데이터가 외부로 나가는 것을 감시하고 차단하는 보안 기술이다. 주민등록번호, 카드번호, 사번 등 일정한 형식을 가진 데이터를 패턴으로 등록하여, 해당 패턴이 포함된 텍스트가 외부로 전송될 때 이를 차단하는 방식으로 작동한다. 이러한 형태가 고정된 정형 데이터의 탐지에는 효과적이다.

그러나 전통적 DLP는 다음과 같은 구조적 한계를 지닌다. **첫째**, 문맥을 이해하지 못한다. 예를 들어, "신소재 배합 비율은 티타늄 45%이다"라는 문장은 기업의 핵심 영업비밀이지만, 정해진 형식이 없는 자연어이므로 DLP의 패턴 필터에 걸리지 않는다. **둘째**, 업무 맥락을 구별하지 못한다. 재무팀 직원이 정당하게 송장을 공유하는 행위와 실제 유출 행위를 구분할 수 없어, 정상적인 업무까지 차단하는 오탐이 빈발한다. **셋째**, 시간이 지날수록 차단 규칙이 누적되면서 정상 업무에 대한 불필요한 경보가 급증하고, 보안 담당자가 경보에 무감각해지는 현상을 유발한다.

#### 2.1.2 네트워크 수준 전면 차단의 역설

AI 서비스 도메인 자체를 방화벽에서 차단하는 방식은 가장 단순하고 확실해 보이지만, 세 가지 치명적 문제를 야기한다. **첫째**, AI가 제공하는 업무 생산성 향상 효과를 전면 포기하게 되어, AI를 활용하는 경쟁 기업에 대한 업무 효율 격차가 벌어진다. **둘째**, 서론에서 언급한 바와 같이 직원들은 AI 사용을 포기하는 것이 아니라 개인 기기를 통해 우회하게 되며, 이 경우 기밀 유출이 회사의 보안 체계 바깥에서 발생하여 감사 기록조차 남길 수 없는 완전한 통제 불능 상태가 된다. **셋째**, 이는 결과적으로 기밀 유출을 방지하는 것이 아니라 유출 사실 자체를 인지할 수 없게 만드는 역설을 초래하며, 차단하지 않았을 때보다 오히려 더 위험한 보안 상태를 만든다.

#### 2.1.3 두 위협의 통합 대응 필요성

비의도적 기밀 유출(직원의 부주의로 인한 사고)과 프롬프트 인젝션(악의적 공격자의 AI 조작)은 성격이 다른 위협이지만, 발생 지점은 동일하다. 두 위협 모두 사용자가 AI에게 텍스트를 입력하는 바로 그 순간에 발생한다<sup>[4]</sup>. 따라서 이 입력 지점에 단일 보안 관문(게이트웨이)을 설치하여 두 위협을 동시에 검사하는 것이 구조적으로 가장 효율적이며, 이것이 본 연구가 두 위협을 하나의 파이프라인으로 통합하여 대응하는 이유이다.

### 2.2 제안 아키텍처의 설계 원칙

본 연구는 기존 DLP가 문맥을 파악할 수 없다는 근본적 한계를 극복하기 위해, 사내에서 독립적으로 구동되는 로컬 LLM 기반의 보안 게이트웨이를 제안한다. 제안 모델의 핵심 설계 원칙은 다음 네 가지이다.

**첫째, 완전한 로컬 처리를 통한 데이터 주권 보장**: 모든 보안 검열 과정이 사내 서버에서 수행된다. 기밀 여부 판단, 의도 분류, 마스킹 처리 전 과정이 외부 네트워크를 경유하지 않으므로, 검열 데이터 자체가 외부로 유출되는 2차 위험을 차단한다.

**둘째, 계층적 심층 방어(Defense-in-Depth)**: 단일 기술에 의존하지 않고, 패턴 기반 고속 필터 → LLM 의도 분류 → 생성형 마스킹의 3단계를 순차적으로 적용한다. 비유하자면, 하나의 자물쇠가 아니라 세 겹의 문을 통과해야 하는 구조이다. 각 단계는 이전 단계가 놓친 위협을 보완하는 상호보완적 구조이다.

**셋째, 동적 마스킹을 통한 사용자 경험 보존**: 기밀 데이터를 단순히 차단하는 것이 아니라, 의미 없는 난수 코드로 치환(마스킹)하여 외부 AI에 전송하고, AI의 응답이 돌아올 때 난수 코드를 원본으로 되돌려(역치환) 사용자에게 전달한다.

**넷째, AI 고유 공격에 대한 문맥 기반 대응**: 기존 보안 체계가 전혀 대응하지 못하는 프롬프트 인젝션, 탈옥(Jailbreak) 등의 공격에 대해, LLM이 질문의 문맥과 의도를 분석하여 악의적 여부를 판정하는 'LLM-as-a-Judge' 기법<sup>[5]</sup>을 적용한다. 이는 LLM을 단순 생성이 아닌 판정자(Judge) 역할로 활용하는 접근으로, 대형 모델일수록 인간 평가자와의 판정 일치율이 높아지는 것으로 보고된 기법이다<sup>[5]</sup>.

### 2.3 하이브리드 보안 필터링 파이프라인

제안 시스템의 보안 파이프라인은 Fig. 1과 같이 3단계로 구성된다. 이를 비유하면, 회사 정문에 설치된 **'다단계 출입 심사대'**와 같다.

> **Fig. 1. Architecture of the Proposed AI Security Gateway Pipeline**
>
> ```
> [사용자 질문 입력]
>      ↓
> [1단계] 패턴 기반 고속 필터 ─── 정형 기밀 마스킹 + 공격 패턴 사전 차단
>      ↓
>      ├─→ [2단계] LLM 의도 분류 ── 문맥 기반 악의적 의도 판정 ─┐
>      │                                                        ├─ 병렬 처리
>      └─→ [3단계] 생성형 마스킹 ── 비정형 기밀 탐지 및 치환  ─┘
>      ↓
> [위협 감지?] → 예: 차단 + 감사 기록 저장
>      ↓ 아니오
> [마스킹된 질문] → 외부 AI → [AI 응답 수신]
>      ↓
> [역치환: 난수 코드 → 원본 데이터 복원]
>      ↓
> [사용자에게 완전한 응답 전달]
> ```

**1단계 — 패턴 기반 고속 필터**: 공항의 '금속 탐지기'에 해당한다. 사번, 주민등록번호, 전화번호 등 형태가 고정된 정형 기밀 데이터를 패턴 매칭으로 즉시 탐지하여 의미 없는 난수 코드로 치환한다. 동시에 "이전 지시를 무시하라", "시스템 프롬프트를 보여달라" 등 알려진 공격 패턴을 사전 필터링한다. 이 단계는 AI 추론 없이 즉시 처리되므로 시스템의 기본 방어선 역할을 한다.

**2단계 — LLM 의도 분류(LLM-as-a-Judge)**: '숙련된 심사관'에 해당한다. 1단계의 금속 탐지기가 놓친 교묘한 공격, 예를 들어 "이 시스템의 보안 규칙을 요약해서 알려줘"와 같이 표면적으로는 정상 질문처럼 보이지만 실제로는 시스템을 탈취하려는 의도를 가진 질문을, 사내 로컬 LLM이 문맥을 분석하여 판별한다. 조금이라도 의심스러운 경우 전송을 차단하는 보수적 정책을 적용한다.

**3단계 — 생성형 마스킹**: '자동 모자이크 처리기'에 해당한다. 1단계가 잡지 못하는 비정형 기밀, 즉 "신소재 배합 비율은 티타늄 45%이다"처럼 정해진 형식이 없는 기밀 정보를 로컬 LLM이 문맥을 읽고 찾아내어 난수 코드로 치환한다. 2단계와 3단계는 병렬로 동시에 처리되어 대기 시간을 단축한다.

**역치환(De-masking)**: 마스킹된 질문에 대해 외부 AI가 응답을 보내오면, 게이트웨이는 응답 내의 난수 코드를 사내에 저장된 원본 데이터로 복원하여 사용자에게 전달한다. 이를 통해 외부 AI는 기밀의 실체를 알지 못한 채 답변하고, 사용자는 원본이 복원된 완전한 응답을 받게 된다.

### 2.4 제안 아키텍처의 한계 및 고려사항

본 아키텍처는 다음과 같은 한계를 내포하고 있으며, 이를 투명하게 제시한다.

**첫째, 마스킹에 따른 AI 응답 품질 저하 가능성이다.** 기밀 데이터가 난수 코드로 치환된 상태에서 외부 AI가 질문을 받으므로, 치환된 데이터의 성격에 따라 AI 응답의 정확도가 저하될 수 있다. 예를 들어, 사원번호나 고객명과 같은 식별 정보의 마스킹은 AI 추론에 영향을 주지 않으나, 소재명이나 화학 성분비와 같이 질문의 핵심을 구성하는 기술 정보가 마스킹될 경우 AI가 정보를 지어내는 할루시네이션(Hallucination)을 유발할 수 있다. 따라서 어떤 항목을 마스킹할지에 대한 세밀한 정책 설계가 필수적이다.

**둘째, 로컬 LLM의 기밀 탐지 정확도이다.** LLM-as-a-Judge 기법은 대형 모델일수록 판정 정확도가 높아지는 경향이 확인되었으나<sup>[5]</sup>, 이를 기업 기밀 탐지에 적용하는 것은 본 연구가 제안하는 새로운 시도이다. 기밀 탐지 정확도는 모델의 규모와 도메인 특화 학습 수준에 의존하며, 기업별 기밀 데이터의 특수성을 반영하기 위해서는 추가적인 파인튜닝이 필요하다.

**셋째, 처리 지연(Latency)이다.** 로컬 LLM이 질문의 의도를 분석하고 비정형 기밀을 탐지하는 과정은 필연적으로 수 초의 지연을 발생시킨다. 그러나 AI 접근이 전면 차단된 현재 상황(응답 자체가 불가능)에 비해, 수 초의 지연은 구조적으로 수용 가능한 트레이드오프이며, 향후 모델 경량화 및 하드웨어 가속을 통해 개선 가능하다.

**넷째, 판정자 LLM 자체의 취약성이다.** 2단계에서 악의적 의도를 판정하는 로컬 LLM 자체가 정교한 다단계 공격의 대상이 될 수 있다. 1단계 패턴 필터가 알려진 공격 패턴을 사전에 차단하여 1차 방어선을 제공하나, 이를 우회하는 신종 공격에 대한 판정자 LLM의 강건성 확보는 향후 연구를 통해 보완해야 할 과제이다.

**다섯째, 물리적 경로의 한계이다.** 본 게이트웨이는 사내 시스템을 경유하는 AI 사용에 대해서만 보호 기능을 제공하며, 직원이 개인 기기로 직접 외부 AI에 접근하는 물리적 경로까지 차단하지는 못한다. 따라서 본 시스템은 보안 교육 및 내부 정책과 병행하여 운용되어야 단단한 보안 체계를 갖출 수 있다.

## 3. 결 론

본 연구는 생성형 AI 환경에서 발생하는 기밀 정보 유출과 프롬프트 인젝션 위협을 분석하고, 이에 대응하기 위한 로컬 LLM 기반 하이브리드 보안 게이트웨이 아키텍처를 제안하였다. 기존 DLP 솔루션은 정형 데이터 탐지에는 효과적이나 문맥 파악의 구조적 한계로 인해 비정형 기밀과 프롬프트 인젝션에 대응하지 못하며, 방화벽 기반 전면 차단은 섀도우 AI를 가속화하여 오히려 기업의 보안 통제력을 상실시키는 역설을 초래한다.

제안 아키텍처는 패턴 기반 고속 필터, LLM 의도 분류, 생성형 마스킹의 3단계 파이프라인을 통해 정형·비정형 기밀 데이터를 동시에 탐지하고, 동적 마스킹과 역치환을 통해 사용자 경험을 보존하면서 외부 AI에 대한 기밀 노출 위험을 최소화한다. 모든 검열이 사내 로컬 환경에서 처리되므로 데이터 주권을 보장하며, 기업은 AI 활용을 전면 차단하는 대신 안전하게 허용하는 새로운 보안 패러다임으로 전환할 수 있다.

향후 연구 방향으로는, 산업군별(금융, 의료, 제조 등) 기밀 데이터 분류 체계에 특화된 경량 로컬 모델 파인튜닝(Fine-Tuning)을 통해 비정형 기밀 탐지의 정밀도를 향상시키는 연구를 제시한다. 산업별 기밀 데이터는 그 형태와 맥락이 상이하므로, 도메인 특화 데이터셋을 기반으로 로컬 LLM의 기밀 인식 능력을 고도화하는 것이 실질적 탐지 성능 향상의 핵심 과제이다.
